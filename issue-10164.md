## Comment Draft for Issue #10164

We built this. Open source, MIT licensed.

**[tenure](https://github.com/tenure-company/tenure)** — Semantic Execution Router for OpenClaw. Every tool call goes through the SER router into a Temporal primitive on the durable timeline.

### What it proves today

**Cron durability.** Temporal Schedule replaces `setTimeout`. We killed the Worker, missed 2 cycles, restarted — all lines present, zero duplicates, sequence numbers correct. OpenClaw's `runMissedJobs` catches up one slot and skips interrupted one-shot jobs entirely (`src/cron/service/ops.ts:107-134`).

**No-duplicate replay.** Tool results are Temporal Activity completions. If the Worker dies after a file write but before the result reaches the next reasoning step, replay returns the cached result from Event History. The file is written exactly once. OpenClaw's tool results sit in an in-memory buffer with a 30-second best-effort flush (`wait-for-idle-before-flush.ts`).

**Budget caps.** The Workflow stops before overspend. The check runs in Workflow logic — deterministic on replay, survives Worker death.

**Circuit breakers.** 3 consecutive failures or 3 identical calls trip the breaker. Pattern state is reconstructed from Activity history on replay.

### The core gap we found

OpenClaw's only dedup mechanism is an in-memory `Map` on the Gateway (`server-runtime-state.ts`). It dies with the process. After SIGKILL + restart, the same idempotency key is accepted again. The test named "agent dedupe survives reconnect" tests WS reconnect to the *same running process*, not process restart.

Temporal's dedup is structural: completed Activities return cached results on replay. No external cache needed.

Re your open questions:

- **Bundled or user-installed?** User-installed — Tenure is an external adapter, so you bring your own Temporal (local dev server or Temporal Cloud), no gateway fork needed.
- **Task queue naming?** `tenure-{agent-id}-{execution-type}` — one queue per agent per execution class so reads don't block behind writes.
- **Workflow versioning with agent updates?** The SER router is the versioning boundary — execution type + primitive selection is decoupled from skill code, so agent updates don't break in-flight Workflows.
